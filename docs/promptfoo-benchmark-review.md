# Promptfoo Benchmark Review for DockSec

Reference document only - no implementation yet. Purpose: understand what makes
promptfoo (~23.5K stars) successful across product, UX, web UI, website, docs, and
community, and map each element to a concrete DockSec opportunity. Reviewed from the
local promptfoo repo checkout and https://www.promptfoo.dev/ on 2026-07-18.

## 1. What promptfoo is

- CLI + library for evaluating and red-teaming LLM applications. TypeScript/Node,
  MIT licensed, now part of OpenAI but still open source.
- Same "wrap a hard security/quality problem in a developer-friendly local tool"
  positioning DockSec has, but executed at much larger scale.
- Scale snapshot: 106 provider integrations, 67 red-team plugins, 32 attack
  strategies, 231 example configs, ~92 provider doc pages, 77 blog posts, full web
  UI, Docusaurus docs site, Helm chart, Docker image, MCP server, GitHub Action.

## 2. Product and CLI surface

Commands (from `src/main.ts` and `src/commands/`): `eval`, `init` (with
`--example <name>` templates), `view` (launches the local web UI), `redteam`
(init/setup/generate/run/report/discover/plugins), `generate` (synthetic
datasets/assertions), `share`, `import`/`export`, `list`/`show`/`delete`, `cache`,
`config`, `auth`, `validate`, `debug`, `logs`, `feedback`, `retry`, `optimize`,
`mcp`, `code-scan`, `model-scan`.

Patterns worth copying:

- **Guided onboarding.** `promptfoo init --example getting-started` scaffolds a
  working project; `redteam setup` opens an interactive web wizard. First success in
  under two minutes, no config authoring required.
- **Results are persisted and browsable.** Every run lands in a local SQLite DB
  (`~/.promptfoo/promptfoo.db`, Drizzle ORM); `list`/`show`/`view` browse history.
  This enables trend tracking, diffing, and the web UI for free.
- **`view` command.** One command opens a rich local web viewer over past results.
  This is the single biggest UX differentiator over plain CLI tools.
- **`share`.** Generates a shareable link for a result - a viral growth loop (every
  shared eval markets the tool).
- **`validate`** for config files, **`retry`** for failed runs, **`feedback`** built
  into the CLI, **`cache`** as an explicit managed subcommand.
- **Config-file-first design.** One `promptfooconfig.yaml` with a published JSON
  Schema (generated by `scripts/generateJsonSchema.ts`), so editors autocomplete it.
- **Library API in addition to CLI** (importable Node package), plus an MCP server
  so AI agents can drive it.

## 3. Web UI (`src/app/`)

- React 19 + Vite + MUI/Tailwind SPA served by a local Express server
  (`promptfoo view`); dev on :3000, API on :15500. Fully local, no cloud required.
- Pages: eval results matrix (side-by-side comparison grid), eval creator, run
  history, datasets, prompts, red-team setup wizard, red-team report dashboard,
  model-audit pages, media viewer.
- Has its own `UI_GUIDELINES.md` (typography scale, composable primitives) - design
  consistency is enforced as a repo convention, which is why the UI looks polished.
- The red-team report is a visual dashboard (severity breakdown, categories,
  remediation) - screenshots of it are used heavily in marketing.

## 4. Website (promptfoo.dev)

Structure observed:

- Hero: sharp value proposition ("Ship agents, not vulnerabilities"), two CTAs
  (Book a Demo / Try Open Source).
- Social proof: Fortune 500 logo wall, download/community stats.
- A three-step "how it works" (Connect, Attack, Fix) with a copy-pasteable command
  in the hero area.
- Persona-based sections (CISO / Security Director / Developer).
- Product pages per capability (red teaming, guardrails, model security, LLM
  vulnerability scanner), solutions pages, pricing, events, press, security/trust
  pages (SOC2, ISO 27001), contact.
- Everything lives in the same monorepo under `site/` (Docusaurus): marketing pages,
  docs, and blog share one build and one design system.

## 5. Documentation (site/docs)

- Docusaurus site with clear top-level areas: getting-started, installation, usage,
  configuration, providers (92 pages, one per integration), red-team, guides,
  integrations, enterprise, model-audit, code-scanning, tracing, FAQ, releases.
- **Guides are SEO machines**: comparison posts ("GPT vs Claude vs Gemini",
  "Deepseek benchmark", "Mixtral vs GPT") and task guides ("evaluate RAG",
  "prevent LLM hallucinations", "LLM as a judge"). These rank on search and funnel
  users into the tool.
- **Compliance-mapping docs**: OWASP LLM Top 10, MITRE ATLAS, NIST AI RMF, EU AI
  Act, GDPR, ISO 42001, DoD AI ethics. Mostly metadata over existing features, but
  extremely persuasive for enterprise evaluators.
- **Integration docs for every CI system**: GitHub Actions, GitLab, Jenkins, Azure
  Pipelines, CircleCI, Bitbucket, Travis, plus Jest/Mocha, n8n, SonarQube, Splunk,
  Langfuse, Helicone, Google Sheets, MCP.
- Active blog (77 posts) with auto-generated OG images.

## 6. Engineering and community practices

- Release automation: release-please (conventional commits drive automated
  releases and CHANGELOG), npm + brew + pip + npx distribution, Docker image
  workflow, Helm chart.
- Renovate for dependency updates, Biome + Prettier, Vitest, Knip dead-code CI
  check, coverage ratchets, architecture-boundary checks, CodeQL-equivalent scans.
- CITATION.cff (academic citations), SECURITY.md with coordinated disclosure,
  contrib.rocks contributor wall in README, Discord community badge front and
  center.
- AGENTS.md/CLAUDE.md in every directory - the repo is deliberately optimized for
  AI-assisted contribution, which lowers the contribution barrier.
- README style: minimal badges, one-line value prop, quick start in the first
  screenful, three screenshots/GIFs (web matrix, CLI, red-team dashboard), "Why"
  bullets, links out to docs. Short - the website carries the depth.

## 7. Gap analysis: DockSec vs promptfoo

What DockSec already has that matches promptfoo patterns: strong CI story
(exit codes, SARIF, baseline, waivers, JSON output), privacy story, GitHub Action,
install-skill, comparison table in README, OpenSSF badge.

Main gaps, in rough order of impact:

1. **No web UI / `docksec view`.** HTML reports exist, but there is no persistent
   results store or local dashboard. Promptfoo's viewer is its most marketable
   surface.
2. **No results history.** Reports are one-shot files; no DB, no trends, no
   diffing between scans (already P2.3 in CLAUDE.md).
3. **No dedicated website.** DockSec has the OWASP project page and README only.
   No landing page with hero, screenshots, copy-paste quick start, persona
   sections, or SEO content.
4. **No docs site.** Everything is in one long README. No guides, no per-scanner
   pages, no compliance-mapping pages (CIS Docker Benchmark, NIST 800-190, OWASP
   Docker Top 10 are the direct analogs).
5. **No interactive onboarding.** No `docksec init` producing a sample vulnerable
   Dockerfile/compose project for a guaranteed first-run success.
6. **No share loop.** Nothing a user can send a teammate except a file.
7. **Thin CI-system coverage** beyond GitHub (already P1.6): no Jenkins/GitLab/
   Azure examples in docs.
8. **No blog / SEO content**: comparison and how-to articles ("Trivy vs Snyk vs
   DockSec", "harden a Python Dockerfile", "docker compose security checklist")
   are the promptfoo growth engine and are missing entirely.
9. **Single distribution channel** (pip). Promptfoo ships npm/brew/pip/npx/Docker/
   Helm. DockSec analogs: brew formula, official Docker image (P1.7), maybe pipx
   note and a uv-friendly install.
10. **No community infrastructure** beyond OWASP Slack: no Discord/discussions
    activity strategy, no contributor wall, no "good first issue" pipeline.

## 8. Recommendations for DockSec, prioritized

### Phase 1 - foundation (high impact, moderate effort)

1. Results history: persist scan results in SQLite under `~/.docksec/` (score,
   counts, findings per image/dockerfile over time). Enables everything below.
2. `docksec view`: local web dashboard over scan history - severity charts, score
   trend, findings table with filters, report download. Start simple (FastAPI or
   even a static-bundle server); this becomes the hero screenshot.
3. `docksec init`: scaffold an example project (vulnerable Dockerfile + compose +
   ignore file + CI snippet) so the first run always finds something interesting.
4. Docs site: MkDocs Material or Docusaurus in a `site/` dir, published via GitHub
   Pages at first. Sections: getting started, CLI reference, CI integrations
   (GitHub/GitLab/Jenkins/Azure), reports, privacy, compliance mapping, FAQ.
5. README slim-down to promptfoo style: fewer badges, value prop + quick start +
   three visuals (terminal GIF of a scan, HTML report screenshot, future dashboard)
   in the first screenful; push depth to the docs site.

### Phase 2 - growth (marketing and reach)

6. Landing page (docksec.dev or the OWASP page rebuilt): hero with one-line value
   prop and a copy-paste command, how-it-works (Scan, Analyze, Fix), screenshots,
   persona angles (developer / platform team / security team), install options.
7. Distribution: official Docker image on GHCR (P1.7), Homebrew formula, GitHub
   Marketplace listing for the Action.
8. Content engine: 1-2 guides per month on the docs site (comparisons, hardening
   how-tos, CIS/NIST 800-190 mapping pages). This is the durable star driver.
9. Share loop: `docksec share` producing a self-contained HTML report link (even
   a gist/artifact-based v1), or at minimum a markdown summary for PR comments
   (P2.4).
10. Demo GIF/screencast in README (promptfoo's self-grading GIF equivalent:
    a scan of a deliberately bad Dockerfile ending in the Quick take).

### Phase 3 - product depth (existing roadmap, promptfoo-validated)

11. Config file `.docksec.yml` (P1.1) with a published JSON Schema for editor
    autocomplete, like promptfooconfig.
12. Compliance/policy packs (P2.5) plus matching docs pages - promptfoo shows the
    docs pages alone carry enterprise weight.
13. K8s/Helm scanning (P2.1), registry scanning (P1.2), trend/delta output (P2.3).
14. MCP server mode (`docksec mcp`) so AI agents can invoke scans - complements
    the existing install-skill feature and is cheap to build over the JSON output.
15. Community: enable GitHub Discussions, label good-first-issues, add a
    contributor wall to the README, consider release-please-style automated
    releases keyed to conventional commits.

## 9. What not to copy

- Promptfoo's enterprise/cloud split, pricing pages, and "book a demo" funnel - not
  applicable to an OWASP vendor-neutral project; keep the "free, local,
  no-telemetry" position as the differentiator.
- Telemetry (`src/telemetry.ts`): promptfoo collects usage data; DockSec's
  "no telemetry" stance is a stated selling point - keep it.
- The full TypeScript monorepo complexity (workspaces, Drizzle migrations, Knip,
  architecture checks) is right-sized for their team, not for DockSec today. Adopt
  incrementally: results DB and docs site first.
